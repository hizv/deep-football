Potential-Based Reward Shaping (PBRS) in Soccer TwosThe Sparse Reward ProblemIn the default Soccer Twos environment, the reward signal is notoriously sparse: agents primarily receive a positive reward when they successfully score a goal, which is mathematically calculated as $1$ minus an accumulated time penalty. Because scoring a goal requires a complex, long-horizon sequence of precise actions, relying solely on this sparse terminal signal makes learning incredibly slow and sample-inefficient.The Mathematical FrameworkTo accelerate learning, we must provide intermediate, dense rewards. However, naively adding arbitrary rewards (such as granting $+0.1$ every time the agent touches the ball) often leads to "reward hacking." In this scenario, the agent might learn to simply vibrate against the ball to accumulate infinite points without ever attempting to score.Potential-Based Reward Shaping solves this by mathematically guaranteeing that the optimal policy of the underlying game remains completely unchanged. The dense shaping reward $F$ is defined as the difference between the calculated potential of the new state $\Phi(s')$ and the old state $\Phi(s)$, discounted by the environment's discount factor $\gamma$:$$F(s, a, s') = \gamma\Phi(s') - \Phi(s)$$This specific telescopic mathematical property ensures that any cyclical or repetitive behavior automatically cancels out its own accumulated rewards, fundamentally preventing exploitation.Defining the Potential FunctionA simple and highly effective potential function $\Phi(s)$ for the soccer environment is the negative Euclidean distance between the ball and the opponent's goal.$$\Phi(s) = -\text{distance}(\text{ball}, \text{goal})$$For example, if the ball starts at a distance of $20$ units ($\Phi(s) = -20$) and the agent's action moves it to a distance of $15$ units ($\Phi(s') = -15$), the shaping reward injected into that specific timestep would be calculated as $\gamma(-15) - (-20)$. This provides a continuous, dense reinforcement signal whenever the agent successfully progresses the ball forward, rather than waiting for the episode to end.Implementation ArchitectureTo implement this cleanly without altering the underlying PPO, SAC, or DQN algorithms, you should wrap the instantiated environment using a custom gymnasium.RewardWrapper.Intercept the Step Data: Override the wrapper's step() or reward() function to intercept the data before it reaches the RL algorithm.Extract Coordinates: The underlying Soccer Twos step() method returns an info dictionary (the fourth element in the tuple) containing critical programmatic metadata. You can extract the exact absolute coordinates using this dictionary (e.g., parsing the ball_state for its position array).Calculate the Difference: Compute the Euclidean distance to the target goal, calculate the new potential $\Phi(s')$, apply the PBRS formula using the stored potential from the previous state $\Phi(s)$, and safely add this new shaped value to the default sparse reward array.
